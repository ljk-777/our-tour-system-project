# 周报生成 Skill

## 目标

生成一份结构完整、如实反映进展的项目周报，并通过"对抗核查"步骤主动暴露计划与实际的差距，而不是美化进展。

---

## 执行步骤

### 第一步：收集本周变更信息

**本地 Git（必做）**
```bash
git log --since="7 days ago" --oneline --all
git log --since="7 days ago" --stat --all | head -100
```

**GitHub MCP（如可用）**
- 调用 `list_commits`，筛选最近 7 天的提交
- 调用 `list_prs`，查看本周合并或开放的 PR

> 如果 GitHub MCP 不可用，仅依赖本地 Git，并在报告开头注明"仅基于本地记录生成"。

---

### 第二步：读取上期周报

读取 `weekly-reports/` 目录下编号最大的周报文件，重点提取：
- **"五、下周计划"** 中列出的所有计划项（这是本次核查的对照依据）
- **"四、当前存在的问题"** 中遗留的问题

---

### 第三步：对抗核查（核心）

这是对抗式编程的关键步骤。在生成报告之前，先做一次"质疑者审查"：

对每条上周计划，逐一判断：
- ✅ 已完成 — git 记录或代码变更能够印证
- ⚠️ 部分完成 — 有进展但未达到计划描述的状态
- ❌ 未完成 — 本周无相关变更记录

对上期遗留问题，同样判断是否已解决。

将核查结果作为**第四节"当前存在的问题"**的输入，确保问题被如实记录，而不是悄悄略过。

---

### 第四步：生成周报

按以下固定结构输出，不要省略任何章节：

```
# 第N周项目周报

## 一、本周项目整体进展
（2-4 句话，概括本周整体方向与核心产出，指出进展的质量而非数量）

---

## 二、本周各模块进展

### 1. [模块名]
（按实际变更分模块描述，每模块 3-6 条 bullet，要具体到文件或功能）

### 2. [模块名]
...

---

## 三、本周每位同学工作进展

### 1. 戴鸿（后端·数据与算法）
### 2. 李佳坤（后端·业务与检索）
### 3. 万子豪（前端·地图与导航）
### 4. 缑文轩（前端·社区与演示）

（每人 2-4 条 bullet，基于 git blame 或提交记录推断，无法确认的注明"待核实"）

---

## 四、当前存在的问题

（直接引用对抗核查结果，列出：
1. 上周计划中未完成或仅部分完成的项
2. 上期遗留问题中仍未解决的项
3. 本周新出现的技术债或风险点）

---

## 五、下周计划

（3-5 条，优先补上本周未完成项，再列新增计划）
```

---

## 输出要求

- 直接输出 Markdown，不要加额外解释
- 周次编号从上期报告自动 +1 推断
- 模块划分跟随 git 提交涉及的目录（`src/backend/`、`src/frontend/`、`.claude/`、`weekly-reports/` 等）
- **第四节"当前存在的问题"不能为空** — 如果对抗核查发现所有计划都完成了，说明计划制定得不够有野心，应在此指出
- 如果无法通过 git 记录确认某人的贡献，使用"待核实"而不是猜测
